Two Powerful Testing Modes

Choose the right approach for your load testing needs — test full user journeys or stress individual endpoints. Both support CSV/Excel template upload or interactive Q&A.

Most Popular
📋

Business Scenario Mode

Multi-step user journeys with correlation between steps

Example Journey
Login → Search → Add to Cart → Checkout → Payment
  • Scenario-based thread groups — each journey runs as a realistic user flow
  • Step-level correlation — extract tokens, IDs, and session data between steps
  • Per-step NFR overrides — different SLAs for Login (1s) vs Payment (3s)
  • Concurrent user distribution — 30% Purchase, 20% Browse, 50% Search
  • CSV/Excel template upload — define all journeys in one file
  • Arrow syntax — Login → Search → Cart → Checkout
Best for: E-commerce flows User onboarding Checkout pipelines Multi-API workflows
High Precision
🎯

Individual API Mode

Test each endpoint independently with its own RPS or Concurrent User %

Example Config
APIMethodLoad
/api/usersGET40% CU
/api/ordersPOST25% CU
/api/productsGET100 RPS
  • Per-endpoint load control — set RPS or Concurrent Users % per API
  • Independent thread groups — each endpoint tested in isolation
  • Flexible load types — mix RPS-based and CU%-based in one test
  • Auto-validation — CU% sums to ~100%, no RPS+CU conflicts
  • CSV/Excel template upload — bulk define all endpoints
  • Method defaults — GET assumed if omitted
Best for: API benchmarking Microservice testing Endpoint capacity SLA validation

Both modes support 6 test types (baseline, load, stress, spike, soak, capacity), JMeter & k6, and all 6 execution modes (local, cloud, distributed, BYOI, distributed BYOI, Azure Load Testing).

How It Works

From commit to confidence — a complete workflow powered by NFR-driven testing, all from Copilot Chat

⚡

Analyze Code & Complexity anytime

Commit as usual, then ask the agent to analyze your diff. It detects performance smells ([PERF]: N+1 queries, await-in-loop, missing caching) and architecture violations ([ARCH]: SRP violations, god classes, circular deps) across any language. Automatically runs cyclomatic & cognitive complexity analysis on changed files. Offers one-click auto-fix patches for every issue found.

1

Provide or Generate Swagger / OpenAPI

Start by pointing the agent to your Swagger/OpenAPI spec. It auto-discovers specs in your workspace. No spec? The agent generates one from your source code using AI-powered semantic analysis — extracts routes, models, auth patterns, and validation rules from any framework. You can also fetch specs live from Azure API Management via the APIM integration (subscription → resource group → APIM instance → API).

2

Choose Testing Mode

Pick one of two modes: Business Scenarios — multi-step user journeys (Browse → View → Purchase) with correlation between steps, or Individual API Testing — test each endpoint independently with its own RPS or concurrent-user share. Upload a CSV/Excel template to skip manual Q&A entirely.

3

Capture NFR & Select Test Type

Define your non-functional requirements — concurrency targets, response time SLAs (p95/p99), throughput, error rate budget, and test duration. Then choose from 6 test types: baseline, load, stress, spike, soak/endurance, or capacity. NFR values drive the load profile, assertions, and pass/fail verdicts downstream.

4

Generate & Sanitize Test Scripts

The agent generates JMeter (.jmx) or k6 (.js) scripts with NFR-driven enhancements — duration assertions, auto-correlation (POST → GET/PUT/DELETE chain extraction), think times, optional App Insights BackendListener, and a complete ASCII tree of every element. Scripts are automatically sanitized and validated before use.

5

Smart Infrastructure Advisor & Change Tracking

Before execution, the agent deep-analyzes your test script — thread counts, sampler complexity, write ratio, payload sizes, SSL usage — and calculates the optimal infrastructure. It also runs structural fingerprinting to compare the current script against the agent-generated baseline, showing exactly what was modified (and ignoring cosmetic changes like whitespace or comments).

5.5

Sanity Test optional

Agent asks: “Would you like to run a quick sanity test locally first?” If yes — runs the same script with 1 thread and 2 iterations (~15 sec). Checks connectivity, auth tokens, and endpoint health. If sanity passes, proceeds to full test. If it fails, shows per-endpoint results and actionable fixes.

6

Execute Tests

Run locally, on a single Azure VM, across a distributed master-slave cluster, on your own infrastructure via SSH (single or distributed BYOI), or on Azure Load Testing (fully managed, no VMs needed) — all from a single chat command. Auth is auto-handled via authconfig.json when configured.

7

Analyze Results, Report & App Insights

Get comprehensive analysis with p50/p90/p95/p99 percentiles, per-endpoint breakdown, throughput, and error rates. Automatically generates a downloadable Markdown report with NFR pass/fail verdicts, infrastructure utilization, App Insights client-vs-server comparison, agent-vs-manual script change tracking, and actionable recommendations. Saved to .perf-output/reports/.

8

Proactive Fix & Re-Test

When tests reveal slow endpoints (p95 > SLA), the agent searches your source code for root causes — N+1 queries, await-in-loop, missing caching — applies targeted fixes with your approval, re-runs the test, and shows before/after improvement. Reuse existing clusters for instant re-runs.

How to Install

VS Code UI

1 Download the .vsix for your platform above
2 Open VS Code → Extensions (Cmd+Shift+X / Ctrl+Shift+X)
3 Click … → Install from VSIX…
4 Select the downloaded file — done!

Command Line

# macOS Apple Silicon
code --install-extension performance-agent-2.0.0-darwin-arm64.vsix

# macOS Intel
code --install-extension performance-agent-2.0.0-darwin-x64.vsix

# Windows x64
code --install-extension performance-agent-2.0.0-win32-x64.vsix

# Linux x64
code --install-extension performance-agent-2.0.0-linux-x64.vsix

After Installation

Open a repo or folder in VS Code — the extension activates per-workspace
On successful installation, an agent file is auto-created at .github/agents/ and 14 skill files at .github/skills/ in your workspace
Open GitHub Copilot Chat (Cmd+Shift+I / Ctrl+Shift+I)
Click the mode dropdown at the top of Copilot Chat — you’ll see a list of available agents. Select “AgentPT.ai” from the list.
Once selected, AgentPT.ai will appear at the bottom of the Copilot Chat panel — you’re good to go!

Download Extension

npm Package

Install the MCP server globally via npm and configure manually.

npm install -g @performance-agent/mcp-server

Then create .vscode/mcp.json:

{
  "servers": {
    "performance-agent": {
      "command": "npx",
      "args": ["@performance-agent/mcp-server"]
    }
  }
}

No API keys needed. Works offline for test generation.

Build from Source

Clone and build for development or contributing.

git clone https://github.com/amanraj-ms/copilot-perf-agent.git
cd copilot-perf-agent/mcp-server
npm install && npm run build

Then create .vscode/mcp.json:

{
  "servers": {
    "performance-agent": {
      "command": "node",
      "args": ["/path/to/mcp-server/dist/index.js"]
    }
  }
}

Troubleshooting

Common issues and solutions

MCP Server not loading / Tools don't appear in Copilot Chat
  1. Verify .vscode/mcp.json exists with the correct absolute path to dist/index.js
  2. Restart VS Code: Cmd+Shift+P → Developer: Reload Window
  3. Check the Output panel: View → Output → select "MCP: performance-agent"
  4. Ensure npm run build completed successfully and dist/index.js exists
  5. Make sure you're in Agent Mode, not Ask or Edit mode
JMX/k6 generation fails
  1. Verify your Swagger file is valid JSON: cat swagger.json | jq .
  2. Ensure the paths object exists and contains endpoints
  3. Both Swagger 2.0 and OpenAPI 3.x formats are supported
  4. Check file encoding is UTF-8 (no BOM)
Azure VM provisioning fails
  1. Login to Azure CLI: az login then az account show
  2. Check subscription quota: Azure Portal → Subscriptions → Usage + quotas
  3. Try a different region or smaller VM size
  4. Review Terraform logs in .perf-output/terraform-logs/
VM metrics return empty
  1. Wait 5–10 minutes after test completes (Azure Monitor delay)
  2. Try a wider time range: time_range: "6h"
  3. Verify the VM is still running: az vm show -g <rg> -n <name>
  4. Ensure your account has "Monitoring Reader" role
Sanity test shows 0 endpoints or empty results
  1. If the JTL file is empty, the test may have timed out before completing. Increase the timeout or check if your API has very slow endpoints (e.g., returning large payloads that take >60s per request).
  2. Verify the JMX overrides applied correctly — the sanity log should show threads=1 and ramp-up=0.
  3. Check the sanity JMeter log at .perf-output/sanity/sanity-jmeter.log for errors.
  4. Ensure the MCP server was rebuilt after any source changes: cd mcp-server && npm run build
APIM integration fails / Cannot list APIM instances
  1. Ensure Azure CLI is logged in: az login
  2. Verify you have API Management Service Reader role on the subscription or resource group
  3. Check that the APIM instance is in a Succeeded provisioning state (not creating or failed)
  4. If using fetchSwaggerFromAPIM in file mode, ensure the exported file is valid JSON
  5. For raw HTTP requests, ensure the request format includes method, URL, and headers
Azure Load Testing (ALT) test fails or shows NOT_APPLICABLE
  1. Verify the Microsoft.LoadTestService resource provider is registered: az provider register --namespace Microsoft.LoadTestService
  2. Ensure the JMX file is valid and runs locally first (use sanity test)
  3. Check that CSV data files referenced in the JMX are included in the upload
  4. If results are empty, the test may have been cancelled — check ALT portal for error details
  5. For ZIP extraction issues, ensure the MCP server is on the latest build
Auth configuration fails / Token not injected
  1. Run analyzeSwaggerAuth first to detect the auth scheme from your OpenAPI spec
  2. Verify authconfig.json has the correct token endpoint URL and grant type
  3. Run validateAuthConfig to check for missing or invalid fields
  4. For OAuth2 client_credentials, ensure client ID and secret are correct (not placeholder values)
  5. Check that the token endpoint is reachable from your network / Azure VM

Test Execution Modes

Choose the right execution mode for your workload — from local smoke tests to distributed load tests

💻

Local

Run JMeter or k6 on your machine. Zero setup if the tool is installed. Great for quick smoke tests and script validation.

  • Auto-detect installed tools
  • Instant execution, no cloud needed
  • JMeter & k6 supported
☁️

Single VM (Azure)

Provision a single Azure VM, run the test, download results, and clean up. Fully automated via Terraform.

  • 7 VM size presets (small to GPU)
  • Auto-provisioning & teardown
  • Correlated VM metrics (CPU, mem, net)
🌐

Distributed (Master + Slaves)

1 master VM + N slave VMs in a private VNet. JMeter distributed mode for heavy workloads (thousands of concurrent users).

  • Auto-scaled slave clusters
  • VNet-based private communication
  • Cluster reuse for rapid re-runs
🔌

BYOI (Bring Your Own)

Run tests on any machine you can SSH into — your own VMs, bare-metal servers, or any cloud provider. Supports single-VM and distributed master-slave execution.

  • Single VM or distributed master + N slaves via SSH
  • Auto-install JMeter if missing
  • Just provide host, user, SSH key
  • Works with any cloud or on-prem server
New
🧪

Azure Load Testing (Managed)

Fully managed cloud performance testing — no VMs to provision, no Terraform needed. Upload your JMX script to Azure Load Testing, execute at scale, and download parsed results with per-API breakdown.

  • Zero infrastructure management
  • Auto-scaling test execution
  • Enterprise-grade result reporting
  • Resource reuse for rapid re-runs
